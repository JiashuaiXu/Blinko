# 设计核心原则

本文档定义 Blinko / 明眸 项目的设计核心原则，强调**精简设计**，避免过度设计。

---

## 1. 核心设计原则

### 1.1 精简优先（Simplicity First）

> **"简单是终极的复杂"** - 达芬奇

- **最小化界面元素**：只显示必要的信息和操作
- **减少认知负担**：用户不需要思考就能理解
- **避免功能堆砌**：只实现核心功能，不添加不必要的特性

**设计决策流程**：
1. 这个功能是否必需？
2. 能否用更简单的方式实现？
3. 能否删除某些元素？

### 1.2 避免过度设计（No Over-Engineering）

**过度设计的表现**：
- ❌ 为未来可能的需求提前设计
- ❌ 添加用户不需要的功能
- ❌ 复杂的交互流程
- ❌ 过多的配置选项
- ❌ 不必要的动画效果

**精简设计的表现**：
- ✅ 只实现当前需要的功能
- ✅ 简单的交互流程
- ✅ 合理的默认值
- ✅ 轻量的动画反馈

## 2. UI 设计原则

### 2.1 最小化界面

**原则**：
- 默认隐藏非必要信息
- 使用渐进式展示（需要时再显示）
- 避免信息过载

**示例**：
```
✅ 精简设计
┌─────────────────┐
│  今日提醒: 5    │
│  休息时长: 20m  │
└─────────────────┘

❌ 过度设计
┌─────────────────────────────┐
│ 今日提醒: 5  │ 本周: 35      │
│ 休息时长: 20m │ 平均: 18m     │
│ 坐姿提醒: 3   │ 眨眼提醒: 2   │
│ [详细统计] [设置] [导出]     │
└─────────────────────────────┘
```

### 2.2 减少视觉噪音

**原则**：
- 使用一致的视觉语言
- 减少颜色数量（最多 3-4 种主色）
- 避免过多的装饰元素
- 留白比填充更重要

**颜色使用**：
- 主色：1-2 种（品牌色）
- 功能色：2-3 种（状态色：正常/警告/错误）
- 中性色：灰色系（文本、边框）

### 2.3 简化交互

**原则**：
- 减少点击次数（目标操作不超过 2 步）
- 使用清晰的按钮标签
- 避免复杂的菜单结构
- 提供快捷操作

**交互层级**：
```
✅ 2 层：托盘图标 → 主面板
❌ 3+ 层：托盘图标 → 菜单 → 子菜单 → 功能
```

## 3. 功能设计原则

### 3.1 核心功能优先

**功能优先级**：
1. **P0（必须）**：定时提醒、系统托盘
2. **P1（重要）**：眨眼检测、坐姿检测
3. **P2（可选）**：数据统计、多语言
4. **P3（未来）**：高级功能、插件系统

**开发原则**：
- 先实现 P0 功能
- P1 功能可以简化实现
- P2 功能可以推迟
- P3 功能暂时不考虑

### 3.2 默认值优先

**原则**：
- 提供合理的默认值
- 减少用户配置负担
- 只在必要时暴露配置选项

**示例**：
```
✅ 好的设计
- 默认提醒间隔：20 分钟
- 默认启用定时提醒
- 默认关闭检测功能（需要用户主动开启）

❌ 过度设计
- 提供 10+ 个配置选项
- 每个功能都需要用户配置
- 复杂的配置向导
```

### 3.3 渐进式增强

**原则**：
- 基础功能必须可用
- 高级功能是可选的
- 不影响核心功能

**示例**：
- 基础：定时提醒（必须）
- 增强：眨眼检测（可选）
- 增强：坐姿检测（可选）

## 4. 代码设计原则

### 4.1 避免过早抽象

**原则**：
- 只有 2 处以上重复时才抽象
- 优先使用简单直接的方式
- 避免过度设计的数据结构

**示例**：
```typescript
// ✅ 精简设计
function showReminder() {
  console.log('该休息了！');
}

// ❌ 过度设计
class ReminderManager {
  private strategy: ReminderStrategy;
  private factory: ReminderFactory;
  private logger: Logger;
  // ... 太多抽象
}
```

### 4.2 简化数据结构

**原则**：
- 只存储必要的数据
- 避免过度嵌套
- 使用简单的数据结构

**示例**：
```typescript
// ✅ 精简设计
interface Config {
  interval: number;
  enabled: boolean;
}

// ❌ 过度设计
interface Config {
  reminder: {
    timer: {
      interval: number;
      enabled: boolean;
      metadata: {
        // ... 太多层级
      };
    };
  };
}
```

## 5. 设计决策流程

### 5.1 添加新功能前的检查清单

在添加新功能前，问自己：

- [ ] 这个功能是否真的需要？
- [ ] 能否用更简单的方式实现？
- [ ] 能否删除某些部分？
- [ ] 是否会影响现有功能？
- [ ] 用户是否真的会使用？

**如果答案有任何不确定，先不做。**

### 5.2 设计评审标准

评审设计时，检查：

- **必要性**：功能是否必需？
- **简洁性**：能否更简单？
- **一致性**：是否符合现有设计？
- **可用性**：用户能否理解？
- **维护性**：是否易于维护？

## 7. 精简设计检查清单

在设计任何功能时，检查：

- [ ] 是否只包含必要元素？
- [ ] 是否减少了用户操作步骤？
- [ ] 是否提供了合理的默认值？
- [ ] 是否避免了过度抽象？
- [ ] 是否删除了不必要的功能？
- [ ] 是否保持了简洁的视觉风格？
- [ ] 是否易于理解和维护？

---

**记住：最好的设计是看起来没有设计，最好的功能是用户感觉不到的功能。**
